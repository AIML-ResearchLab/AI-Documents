# Planning

# Planning (Agentic AI Cognitive Pattern)

After the **Goal is defined**, the agent must decide **how to achieve the goal**.

This process is called **Planning**.

`Planning = Breaking the goal into ordered steps or tasks the agent will execute.`

- It converts a `high-level objective → actionable steps`.

## Role of Planning in the Agent Flow

```
Goal
 ↓
Planning
 ↓
Reasoning
 ↓
Validation
 ↓
Action
```

**Planning answers:**

- What steps should be taken?
- In what order?
- What tools or resources may be needed?


## What Planning Produces

A `plan` usually contains:

| Element      | Meaning                     |
| ------------ | --------------------------- |
| Steps        | Tasks to perform            |
| Order        | Execution sequence          |
| Dependencies | Tasks that depend on others |
| Tools        | APIs or commands needed     |
| Conditions   | When steps should execute   |


## Example 1 — Simple Agent (Weather Query)

**Goal:**

`Get current weather in Bangalore`

**Plan:**

```
1 Identify weather data source
2 Call weather API
3 Format response
4 Return answer
```

**Flow:**

```
Goal
 ↓
Planning
 ↓
Tool Action
```

## Example 2 — Incident Remediation Agent

**Goal:**

`Reduce API latency below 200ms`

## Plan Generated by Agent

```
1 Check API logs
2 Check database latency
3 Check CPU utilization
4 If DB overloaded → scale DB
5 If CPU high → scale service
```

This plan allows the agent to **systematically diagnose the issue**.


## Types of Planning in Agent Systems

| Planning Type         | Description                  | Example                    |
| --------------------- | ---------------------------- | -------------------------- |
| Static Planning       | Plan generated once          | Deployment pipeline        |
| Dynamic Planning      | Plan adapts during execution | Incident troubleshooting   |
| Hierarchical Planning | Multi-level planning         | Strategy → tasks           |
| Reactive Planning     | Plan changes based on events | Autoscaling                |
| Multi-Agent Planning  | Multiple agents contribute   | Security + DevOps planning |


## How Planning Uses Reasoning
When the agent receives a **goal**, the planner internally performs reasoning such as:

- Understand the goal
- Identify required tasks
- Determine dependencies
- Order the steps
- Produce the final plan


This reasoning process is often called **Plan Reasoning** or **Deliberative Planning**.

**Flow:**

```
Goal
 ↓
Reason about the goal
 ↓
Break goal into tasks
 ↓
Order tasks
 ↓
Generate plan
```

# Two Planner Architectures

**In advanced agent systems the planner can call tools before creating the plan**, but **not always**. There are **two common planner designs** in real systems.

Understanding this is important because it affects **accuracy**, **cost**, and **latency**.


| Planner Type    | Tool Usage Before Planning | Where Used        |
| --------------- | -------------------------- | ----------------- |
| Static Planner  | ❌ No tools used            | simple agents     |
| Dynamic Planner | ✅ Tools used first         | enterprise agents |


## Static Planner (Simpler)

Planner only uses:

- Goal
- Prior knowledge (LLM training)
- Memory
- Runbooks

**Flow:**

```
Goal
 ↓
LLM Planner
 ↓
Plan
```

**Example**

Goal: `Fix API latency`

Planner generates:

```
1 Check logs
2 Check DB
3 Restart service
4 Validate latency
```

Tools are **not used until execution**.

This works for **simple problems**, but may produce **incorrect plans** because it lacks real-time data.


## Dynamic Planner (Enterprise Approach)

In real incident automation systems, planners usually **collect context first using tools**.

**Flow:**

```
Goal
 ↓
Planner
 ↓
Context tools
 ↓
Planner reasoning
 ↓
Plan
```

This is sometimes called:

- **Context-aware planning**
- **Planning with tool retrieval**

**Incident:**

**Goal generated:**

## 1️⃣ Static Planner

**Concept**
The planner **creates a full plan once** using the goal and prior knowledge.
It **does not gather runtime data before planning**.

The plan is generated **only using LLM reasoning** or **predefined runbooks**.

A **rule-based (static) planner does not “use rules as knowledge to create a plan.”**
Instead it **uses rules to select an existing plan (runbook)**.

So the process is:

`Incident → Rule → Runbook → Plan`

The **plan already exists**, and the rules just **decide which plan to use**.

**Note:** `Static Planner is essentially rule-based.`
It relies on **rules that map incidents → runbooks → steps**.

But in practice, it is usually a **combination of three rule-based mechanisms**:

1️⃣ Pattern matching rules
2️⃣ Incident classification rules
3️⃣ Runbook selection rules

## 1. Rule-Based Static Planner Concept

The planner does **not think or reason**.
It simply **matches the issue to a predefined procedure**.

**Flow:**

```
Incident
 ↓
Rule Engine
 ↓
Runbook Selection
 ↓
Plan Output
```

## 2. Example Rule-Based Logic

Incident:

`Database connection pool exhausted`

Rule:

```
IF issue contains "connection pool exhausted"
THEN use runbook DB_CONNECTION_POOL_EXHAUSTED
```

Planner output:

```
1 Check database connection metrics
2 Identify long-running queries
3 Restart application pods
4 Verify connection pool usage
```

## 3. Example Rule Table

| Rule ID | Condition                                   | Runbook                      |
| ------- | ------------------------------------------- | ---------------------------- |
| R1      | issue contains "connection pool exhausted"  | DB_CONNECTION_POOL_EXHAUSTED |
| R2      | issue contains "Kafka publish failed"       | KAFKA_PUBLISH_FAILURE        |
| R3      | issue contains "Redis cache timeout"        | REDIS_CACHE_TIMEOUT          |
| R4      | issue contains "Unauthorized token missing" | AUTHENTICATION_FAILURE       |


## 4. Runbook Library

Each runbook contains the plan.

Example:

```
{
  "runbook_id": "DB_CONNECTION_POOL_EXHAUSTED",
  "steps": [
    "Check DB connection metrics",
    "Check long-running queries",
    "Restart application pods",
    "Verify DB connection usage"
  ]
}
```

## 5. Example Static Planner Code

**Rule engine**

```
def classify_issue(description):
    desc = description.lower()

    if "connection pool exhausted" in desc:
        return "DB_CONNECTION_POOL_EXHAUSTED"
    if "kafka publish failed" in desc:
        return "KAFKA_PUBLISH_FAILURE"
    if "redis cache timeout" in desc:
        return "REDIS_CACHE_TIMEOUT"

    return "UNKNOWN"
```

**Runbook database**

```
RUNBOOKS = {
    "DB_CONNECTION_POOL_EXHAUSTED": [
        "Check DB connection metrics",
        "Check long-running queries",
        "Restart application pods",
        "Verify DB connection usage"
    ]
}
```

**Static planner**

```
def static_planner(issue_description):
    issue_key = classify_issue(issue_description)

    steps = RUNBOOKS.get(issue_key)

    return {
        "issue_key": issue_key,
        "plan": steps
    }
```

## 6. Example Execution

Input incident:

`Database connection pool exhausted`

Planner output:

```
{
  "issue_key": "DB_CONNECTION_POOL_EXHAUSTED",
  "plan": [
    "Check DB connection metrics",
    "Check long-running queries",
    "Restart application pods",
    "Verify DB connection usage"
  ]
}
```

No LLM used.

## 10. Modern Enterprise Approach

Most companies use **Hybrid Planning**.

Flow:

```
Incident
 ↓
Rule-based planner
 ↓
Known issue?
 ├ Yes → Runbook execution
 └ No → AI planner (LLM)
 ```


 This gives **both safety and intelligence**.

## 1. What the Planner Actually Does

A static planner performs two things:

**1. Identify the incident type**
**2. Select the correct runbook**

It does **not generate steps dynamically**.

**2. Example**

Incident:

`Database connection pool exhausted`

Rule engine checks rules:

| Rule | Condition                                  | Action                        |
| ---- | ------------------------------------------ | ----------------------------- |
| R1   | issue contains "connection pool exhausted" | use runbook DB_POOL_EXHAUSTED |


The planner selects the runbook.

## 3. Runbook (Predefined Plan)

```
{
  "runbook_id": "DB_POOL_EXHAUSTED",
  "goal": "Restore database connectivity",
  "steps": [
    "Check database connection metrics",
    "Check long running queries",
    "Restart application pods",
    "Verify connection pool usage"
  ]
}
```

Planner output becomes:

```
Goal: Restore database connectivity for user-api

Plan:
1 Check DB metrics
2 Check long-running queries
3 Restart application pods
4 Verify DB connections
```

Notice:

- Planner **did not invent steps**
- It **retrieved them from the runbook**

## 4. Static Planner Architecture

```
Incident
 ↓
Incident classifier
 ↓
Rule engine
 ↓
Runbook lookup
 ↓
Plan
```

## 5. Example Static Planner Code

```
RUNBOOKS = {
    "DB_POOL_EXHAUSTED": [
        "Check DB metrics",
        "Check long-running queries",
        "Restart application pods",
        "Verify connection pool"
    ]
}

RULES = {
    "connection pool exhausted": "DB_POOL_EXHAUSTED"
}

def planner(issue_description):
    for pattern, runbook in RULES.items():
        if pattern in issue_description.lower():
            return RUNBOOKS[runbook]

    return ["Escalate incident"]
```

## 6. Difference Between Static Planner and AI Planner

| Feature       | Static Planner      | AI Planner    |
| ------------- | ------------------- | ------------- |
| Plan creation | No                  | Yes           |
| Rules used    | To select runbook   | As context    |
| Steps source  | predefined runbooks | LLM reasoning |
| Adaptability  | Low                 | High          |

## 2️⃣ Context-Aware Planner

**Concept:**

Planner **first gathers real-time system context** using tools, then creates the plan.

This avoids incorrect plans.

**Flow:**

```
Goal
 ↓
Planner
 ↓
Context Retrieval Tools
 ↓
Planner reasoning
 ↓
Plan
```

**Example (Incident Auto-Remediation)**

Goal:

`Restore database connectivity for user-api`

Planner calls tools:

```
query_metrics()
query_logs()
query_db_sessions()
```

Collected context:

```
DB connections = 100%
API latency = 2400ms
```

Planner generates plan:

```
1 Inspect DB sessions
2 Identify idle connections
3 Restart user-api pods
4 Verify DB connections below threshold
```

**Characteristics**

| Feature                | Behavior |
| ---------------------- | -------- |
| Planning time          | medium   |
| Tool calls before plan | ✅ yes    |
| Accuracy               | high     |
| Latency                | moderate |
| Complexity             | medium   |


**When Used**

- incident remediation
- infrastructure troubleshooting
- dynamic environments


## 3️⃣ Iterative Planner

**Concept:**

Planner **creates a partial plan, observes results, then updates the plan**.

Planning becomes a **continuous loop**.


**Flow:**

```
Goal
 ↓
Initial Plan
 ↓
Execute Step
 ↓
Observe Result
 ↓
Refine Plan
 ↓
Next Step
```


**Example**

Goal:

`Resolve database latency`

Initial plan:

```
1 Check DB metrics
2 Restart service
```

After step 1 observation:

```
DB CPU = 90%
Connections normal
```

Planner updates plan:

```
1 Scale DB instance
2 Monitor latency
```

**Characteristics:**

| Feature        | Behavior        |
| -------------- | --------------- |
| Planning style | adaptive        |
| Tool calls     | during planning |
| Accuracy       | very high       |
| Latency        | slower          |
| Complexity     | high            |


**When Used:**

- complex incidents
- research agents
- debugging agents
- autonomous systems

## 4️⃣ Multi-Agent Planner

**Concept:**

Planning is **distributed across multiple specialized agents**.
Each agent contributes expertise.

**Flow:**

```
Goal
 ↓
Planner Agent
 ↓
Specialist Agents
 ↓
Plan Aggregation
 ↓
Execution Plan
```

Goal:

`Restore user-api health`

Agents involved:

| Agent                | Role               |
| -------------------- | ------------------ |
| Infrastructure Agent | analyze Kubernetes |
| Database Agent       | analyze DB         |
| Application Agent    | analyze logs       |
| Security Agent       | validate policy    |


Each provides steps:

```
DB agent → inspect DB connections
Infra agent → restart pods
App agent → analyze error logs
```

Final plan:

```
1 Inspect DB sessions
2 Restart application pods
3 Verify API latency
```

**Characteristics:**

| Feature               | Behavior    |
| --------------------- | ----------- |
| Planning intelligence | distributed |
| Accuracy              | very high   |
| Latency               | slower      |
| Complexity            | very high   |



**When Used**

- enterprise AI operations
- large distributed systems
- autonomous DevOps
- multi-domain reasoning
